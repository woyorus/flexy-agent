# Docs Organization Rules (Draft — for review)

Instructions for the coding agent (Claude Code) on how to organize and maintain the `docs/` folder. Goal: a new agent instance reads CLAUDE.md alone and knows exactly which doc to load — without polluting its context with everything else.

Inspired by [OpenAI's harness engineering approach](https://openai.com/index/harness-engineering/): treat CLAUDE.md as **the table of contents**, not the encyclopedia. Progressive disclosure — the agent starts with a small entry point and is taught where to look next.

---

```
│   └── recipes.md           ← Recipe format, generation constraints, scaling, meal types.
├── design-docs/
│   ├── index.md             ← Catalog of design decisions with status.
│   └── (future design docs) ← One file per significant design decision when needed.
├── plans/
│   ├── 001-calorie-budget-redesign.md
│   └── ...

```

## Rules

### 1. CLAUDE.md is the map, not the manual

CLAUDE.md stays under ~100 lines of prose. It contains:
- **Coding conventions** — rules that apply to every change.
- **Debug workflow** — how we iterate (read logs, diagnose, fix).
- **Docs index** — one table mapping every doc to its purpose and when to load it.

CLAUDE.md must NOT duplicate content from other docs. **Test:** if content exists in both CLAUDE.md and a doc, delete it from CLAUDE.md and leave a pointer.

### 2. Product specs split by concern

A monolithic SPEC.md forces an agent to load 900+ lines when it only needs one flow. `product-specs/` splits by concern with an `index.md` entry point.

Each spec file:
- Starts with a scope line: `> Scope: [what this covers]. See also: [related docs].`
- Stays under ~300 lines. Split further only when exceeded.
- Reflects the **actual code behavior**, not aspirational design. When splitting from the old SPEC.md, verify every claim against the current codebase.

**Known divergences to fix during split** (from code audit):
- Fun food model → completely replaced with flex slots/flex budget system
- Plan flow steps → now includes inline recipe gap resolution phases
- Plan proposer sub-agent → exists in code but not documented in spec
- Breakfast recipes → now use structured components like lunch/dinner
- Fun food user input step → removed; flex slots are auto-generated by plan-proposer

### 3. PRODUCT_SENSE.md is the belief system

Renamed from PROJECT.md. Contains vision, principles, and core beliefs — the "why" behind product decisions. This is the tiebreaker doc when making tradeoffs.

Stays as one file. Don't fragment it until it exceeds ~400 lines.

### 4. Design docs capture significant decisions

`design-docs/index.md` catalogs design decisions. Individual design doc files are created when a decision:
- Changes how a core system works (solver, flows, data model)
- Involves a non-obvious tradeoff worth recording for future context

We don't need this directory populated yet — it's scaffolding for when the product grows. Start with just the `index.md` that explains when to create a design doc.

### 5. BACKLOG.md is the roadmap

Extract the backlog (SPEC.md Sections 9-10) into a standalone `BACKLOG.md` at docs root. Contains:
- What the current version delivers
- What's explicitly out of scope and why
- Versioned feature roadmap (v0.0.x → v0.1.0)

This keeps product-specs/ focused on "what IS" and BACKLOG.md focused on "what's NEXT."

### 6. Plans are execution artifacts, not specs

Plans live in `docs/plans/` with numeric prefixes. They are working documents for multi-step changes.

- Completed plans get a `Status: completed` header line. Don't delete them — they serve as decision history.
- Plans are NOT product specs. If a plan produces lasting product decisions, promote those into the relevant spec or design doc.

### 7. Docs index in CLAUDE.md is the single navigation source

The docs index table in CLAUDE.md lists every doc with:
- File path
- One-line purpose
- When to reference it

**When adding a doc:** update the index in the same commit.
**When deleting or moving a doc:** update the index in the same commit.

### 8. Update docs with the code

When modifying code that changes product behavior, update the relevant doc in the same commit:
- Flow logic → `product-specs/flows.md`
- Solver changes → `product-specs/solver.md`
- Data model changes → `product-specs/data-models.md`
- UI patterns → `product-specs/ui.md`
- Architecture → `ARCHITECTURE.md`
- Product direction → `PRODUCT_SENSE.md`

### 9. Every doc states its scope

Every doc starts with:
```markdown
# Title
> Scope: [what this doc covers]. See also: [related docs].
```

This lets an agent verify it loaded the right file before reading further.

## What we skip from the article (too heavy for v0.0.3)

- **Doc-gardening CI/linters** — no automated freshness checks yet.
- **Quality scoring** — no QUALITY_SCORE.md or domain grading.
- **Generated docs** — we don't auto-generate schema docs.
- **References directory with LLM docs** — we use 3 APIs. Not enough to justify.
- **Top-level concern files** (SECURITY.md, RELIABILITY.md, FRONTEND.md) — overkill for prototype.
- **Agent-to-agent review** — one human, one agent.
